# 要件定義書 — 町内匿名捜査室

## インテント分析

- **ユーザーリクエスト**: idea.md に記載の「町内匿名捜査室」を AI-DLC プロセスで再構築する
- **リクエスト種別**: 新規プロジェクト（Greenfield）
- **スコープ見積もり**: 複数コンポーネント・システム全体（Phase 1: 7 Unit、Phase 2: 2 Unit）
- **複雑度見積もり**: 複雑（AI 統合・行政向けダッシュボード・匿名性設計・ゲーミフィケーション）

## プロダクトビジョン

### 一行ピッチ
探偵ナイトスクープ的な「街の小さな依頼」を、ポケモンGO的に歩いて解きにいくサービス。
入口は「匿名捜査員」。続けるうちに、街から名指しで頼まれる「特命捜査員」になってしまう。
扱うのは、町の小さな違和感を捜査するゲームである。

### プロダクトの核
町内匿名捜査室は、街の中にある「まだ事件になっていない違和感」を、AI が未解決案件のような「事件」として再構成し、市民を匿名捜査員として街へ送り出すサービスである。

**二重構造**:
- 表: 街歩き謎解き・現場検分・証拠写真・推理
- 裏: 投稿観察データを AI が構造化し、行政が使える「確認候補地点」として整理

### ハッカソンテーマへの応答
AWS Summit Japan 2026「人をダメにするサービス」に対して:
- 散歩が現場確認になる
- 雑談が聞き込みになる
- 昔の記憶が証拠になる
- 自分の勘が街の役に立ったと知ると、次の事件も追いたくなる

## 機能要件

### R-01 事件の提供
- MVP では事件 #T-01「誰もぶつかっていないのに、みんな少しだけ身構える角」を固定モックとして提供
- 将来的に Bedrock による動的生成へ切り替えられる構造を維持する
- Event モデルの必須フィールド: `event_id`, `case_number`, `title`, `description`, `public_summary`, `investigation_question`, `choices`, `area_tag`, `status`, `safety_level`, `created_at`, `updated_at`

### R-02 事件一覧表示
- ユーザーが新着事件・進行中事件・自分が関与した事件を確認できる
- MVP では #T-01 が一覧に表示される

### R-03 事件詳細確認
- 事件タイトル・説明・違和感の焦点・他の捜査員の公開可能な投稿断片・次の一手を表示
- 他の捜査員の投稿は一文報告・選択式回答・危険要因タグ程度に限定（個人特定情報なし）

### R-04 捜査報告の投稿
- テキストメモ・写真・選択式推理回答を投稿できる
- 「写真1枚または一文テキストのどちらか1つ」でも成立する
- 推理選択肢はスキップ可能
- **写真付き投稿のフロー（確定）**: draft 作成 → S3 アップロード → finalize の3ステップ
  1. `POST /reports`（draft）: テキスト・選択肢を保存し `report_id` と `upload_url` を返す（`status: draft`）
  2. クライアントが Presigned URL で S3 へ直接アップロード（Exif 除去済み）
  3. `POST /reports/{report_id}/finalize`: アップロード完了を通知し `status: submitted` に更新、ワークフローを起動する
- **テキストのみ投稿のフロー**: `POST /reports`（`status: submitted` で即時確定）→ ワークフロー起動
- ワークフローは必ず `status: submitted` の Report に対してのみ起動する

### R-05 AI による投稿整理
- 投稿を AI（Amazon Bedrock）が要約・分類し、事件単位で整理する
- 出力: 要約・危険要因分類・重複統合候補・次の確認ポイント・行政向け摘要・`has_high_signal` フラグ

### R-06 地点別観察統合
- 同地点・類似地点の報告を `LocationObservation` としてまとめる
- MVP では `location_key = {event_id}#{choice_value}` を使った簡易統合

### R-07 役立っている通知
- 「投稿整理完了通知」と「有効観察通知」の2種類を MVP で実装
- **特命予兆通知（`special_candidate` 遷移時）も MVP で実装する**（通知文のニュアンスのみ・UI 上の役職表示なし）
- `role_state` の DB フィールドは最初から保持し、ワークフロー内で遷移評価を行う
- 通知例:
  - anonymous: 「あなたの報告は、有力な現場手がかりとして整理されました」
  - trusted_candidate: 「あなたの観察は、この種の事件で特に精度が高い傾向があります」
  - special_candidate: 「街は、あなたのような観察者を必要とし始めています。類似案件の追加検分依頼が届く可能性があります」

### R-08 行政ダッシュボード
- 確認候補地点の一覧・詳細確認
- 表示項目: 地点ラベル・事件名・写真・危険要因タグ・要約・時間帯傾向・同種報告件数・優先度分類・レビュー状態
- 行政担当者がレビュー状態（確認済み・現地確認予定・対応不要）を更新できる
- 認証: **2層構造**（UI 保護 + API 保護）
  - Layer 1: Next.js 側の固定パスワード（画面アクセス保護）
  - Layer 2: AdminFunction / DemoFunction 側での共有トークン検証（API 直接アクセス保護）
  - `Authorization: Bearer {ADMIN_TOKEN}` ヘッダーを Lambda で検証し、不一致は 401 を返す
  - ADMIN_TOKEN は環境変数で管理（Next.js・Lambda 間で共有）

### R-09 匿名捜査員として参加開始
- 実名・過度なプロフィール入力なしで事件に参加できる
- ユーザーID: メールアドレスまたは localStorage UUID をクライアント側で SHA-256 ハッシュ化
- ハッシュ化前の値はサーバに保存しない

### R-10 特命への進化予兆
- MVP で実装する（通知文のニュアンスのみ・UI 上の役職名・バッジ表示なし）
- DB 上は `role_state`（`anonymous` / `trusted_candidate` / `special_candidate`）を保持
- ワークフロー内で毎回遷移評価を行い、遷移時に予兆通知を生成する
- 正式昇格・辞令発行は MVP 外

### R-11 デモ用即時処理トリガー
- `POST /demo/trigger`（`Authorization: Bearer {DEMO_TOKEN}`）
- Step Functions Express Workflow の同期実行
- 10秒以内の完了を目標
- **トークン分離**: AdminFunction は `ADMIN_TOKEN`、DemoFunction は `DEMO_TOKEN` で独立管理
  - ADMIN_TOKEN: 行政ダッシュボード API の保護（地点情報・レビュー更新）
  - DEMO_TOKEN: デモ実行権限の保護（ハッカソン審査員向け）

### R-12 投稿の公開可否フラグ
- Report モデルに `visibility_status`（`public` / `hidden` / `deleted`）を保持
- MVP では非公開化 UI は不要だが、データモデル上のフラグは最初から持つ

### R-13 投稿モデレーション状態
- Report モデルに `moderation_status`（`pending` / `approved` / `flagged` / `rejected`）を保持
- MVP では UI 注意喚起 + 運営が手動確認できる `moderation_status` フラグのみ（画像自動解析は MVP 外）

## 非機能要件

### N-01 低コスト・低運用負荷
- サーバレス中心（Lambda・API Gateway・DynamoDB・S3）
- AI 推論は事件生成時と投稿整理時に限定
- 画像解析・音声解析は MVP 外

### N-02 高齢層でも使いやすい UI
- 文字サイズ 16px 以上
- 1画面1目的
- 投稿完了まで3タップ以内
- スマホ中心
- 匿名捜査員としてすぐ事件へ入れる

### N-03 リアルタイム性を過剰に要求しない
- AI 整理・地点統合・通知生成・ダッシュボード更新は通常バッチ（1日1〜2回）
- デモ時のみ即時トリガーで対応

### N-04 監査可能性
- 投稿・AI 要約・通知生成・行政レビュー操作の履歴を追える
- ReviewLog・AISummary・CloudWatch Logs で追跡可能

### N-05 安全配慮 UI
- 投稿前に安全ガイドラインを表示（車道に出ない・危険な場所から撮影しない・顔や車両ナンバーが写った写真は投稿しない・個人を特定しない・私有地に入らない）

### N-06 Unit 単位で実装しやすいこと
- 責務を Unit of Work 単位に分ける

### N-07 プロンプトをコードと分離する
- プロンプトは `prompts/` 配下にファイルとして管理
- `prompts/event-generation/`, `prompts/guardrails/`, `prompts/summarization/`, `prompts/tagging/`, `prompts/notifications/`

### N-08 プライバシー保護
- 実名登録を必須にしない
- ハッシュ化前のメールアドレス・デバイス値はサーバに保存しない
- 行政ダッシュボードには個人単位の行動履歴を表示しない
- 顔や車両ナンバーが明瞭な画像は公開しない
- 写真の Exif 情報をクライアント側 JS で除去してから Presigned URL でアップロード（**確定: 案A**）
- 常時位置情報は取得しない

### N-09 レスポンス時間
- 事件一覧表示: 3秒以内
- 投稿受付完了: 3秒以内
- 行政ダッシュボード表示: 5秒以内
- デモ用即時処理: 10秒以内

## 技術スタック決定事項

### フロントエンド
- Next.js + React + TypeScript
- デプロイ: **AWS Amplify**（決勝デプロイ向け・git push で完結）

### バックエンド
- Amazon API Gateway + AWS Lambda（TypeScript）
- インフラ管理: **AWS SAM でプロトタイプ → AWS CDK TypeScript へ移行**

### ワークフロー
- AWS Step Functions（Express Workflow）

### AI
- Amazon Bedrock（モデル名・リージョンは Construction フェーズ開始前に確認）
- 投稿要約・分類・通知文生成: 軽量モデル優先
- 事件生成: 上位モデル検討

### データ
- Amazon DynamoDB
- Amazon S3（写真保存）

### ローカル開発
- Next.js: `localhost:3000`
- DynamoDB Local: `localhost:8000`（Docker）— **フロントエンド開発・シードデータ確認用**
- Lambda/Bedrock: AWS 実環境を使用
- **シードデータの投入先**: DynamoDB Local（フロントエンド開発時）と AWS DynamoDB（統合テスト・デモ準備時）の両方に対応するスクリプトを用意する
- Lambda は AWS 実環境のため、DynamoDB Local は Lambda からは参照しない（フロントエンド開発専用）

### モニタリング
- Amazon CloudWatch

## 確定した設計決定事項

| 決定事項 | 採用案 | 理由 |
|---|---|---|
| Exif 除去方式 | クライアント側 JS（案A） | 実装コスト低・MVP 向き |
| インフラ管理 | SAM → CDK 移行 | ハッカソン速度 + 長期保守性 |
| 行政ダッシュボード認証 | 2層構造（Next.js 固定PW + ADMIN_TOKEN / DEMO_TOKEN 分離） | API 直接アクセスを防ぐ・権限を分離する |
| Unit 5+6 統合・Unit 7+8 統合 | Phase 1 に7 Unit として含める | デモ完全性を優先・実装シンプル |
| 画像モデレーション | UI 注意喚起 + moderation_status フラグ | MVP 範囲内 |
| 通知種別 | 整理完了通知 + 有効観察通知 + 特命予兆通知（MVP で全実装） | デモで進化予兆を見せる |
| Bedrock モデル確認 | Construction 開始前に確認 | モデル提供状況は変動する |
| セキュリティ拡張 | スキップ（ハッカソン MVP） | PoC・プロトタイプ向け |
| PBT 拡張 | スキップ | 統合テスト・プロンプト評価を優先 |
| フロントエンドデプロイ | AWS Amplify | AWS 統一・git push デプロイ |
| ローカル開発 | DynamoDB Local のみ | Lambda は AWS 実環境 |
| デモ対象者 | ハッカソン審査員 | 技術的深さと完成度を重視 |

## MVP スコープ

### MVP で実装するもの
- 事件 #T-01 のモック固定配信
- 匿名捜査員向けフロントエンド（事件一覧・詳細・投稿・通知）
- 投稿受付・S3 保存（Exif 除去付き）
- Bedrock による投稿整理（要約・危険要因分類）
- 地点別観察統合（最小版）
- 役立っている通知（整理完了 + 有効観察）
- 行政ダッシュボード（確認候補地点一覧・詳細・レビュー状態更新）
- デモ用即時処理トリガー
- `role_state` データモデル（UI 表示は MVP 外）
- `visibility_status` / `moderation_status` フラグ（UI は MVP 外）

### MVP でやらないもの
- リアルタイム位置追跡
- 自動施策決定
- 正式な特命捜査員昇格・辞令発行
- LINE 連携
- Cognito 等の本格認証基盤
- 投稿非公開化 UI
- 運営承認フロー UI
- 画像自動解析（Bedrock Vision）
- 本番自治体データとの直接接続

## 安全・倫理方針（要約）

1. 個人ではなく場所を見る
2. 犯罪ではなく違和感を扱う
3. 表は謎解き、裏は観察データ
4. AI 生成事件は承認制にする
5. 投稿は匿名・最小限・非公開化可能にする
6. 行政には地点・要因単位で返す
7. 利用者には無理な現地行動を求めない

### 禁止する事件・投稿内容
- 特定個人の行動・監視・晒し
- 犯人探し・犯罪捜査を連想させる演出
- 特定店舗・施設への一方的な悪評収集
- 顔や車両ナンバーが明瞭な写真の公開

## 主な利用者像

### 匿名捜査員（代表: 佐藤さん、68歳）
- 退職後3年・スマホは普通に使える
- 地域を歩く習慣がある・昔の道や店の変化を覚えている
- 「ここ、なんか変だ」は感じるが、伝える場を持っていない
- ターゲット層: 主に 60〜75歳

### 行政担当者
- 町村〜小規模市規模の交通安全または生活道路管理担当係員
- 課長級・議会説明・複数課横断は MVP 外
