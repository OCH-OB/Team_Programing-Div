# ユーザーストーリー — 町内匿名捜査室

ブレイクダウンアプローチ: **ペルソナベース × ユーザージャーニーベースのハイブリッド**

---

## 匿名捜査員（佐藤さん）のストーリー

### US-01: 匿名で参加開始する

As a 匿名捜査員,  
I want to 実名や複雑な登録なしにすぐ事件に参加したい,  
So that 最初のハードルが低く、気軽に始められる。

**Acceptance Criteria:**
- [ ] メールアドレスまたはデバイス識別子だけで参加できる
- [ ] 入力した値はクライアント側で SHA-256 ハッシュ化され、元の値はサーバに保存されない
- [ ] 1〜2画面のオンボーディング（サービスの説明・安全ガイドライン）を経てから事件一覧へ進む
- [ ] オンボーディングは30秒以内に完了できる
- [ ] 登録完了後、事件一覧が表示される

*デモ対応: Scene 0（導入）・Scene 1（事件配信）の前提*

---

### US-02: 事件一覧を確認する

As a 匿名捜査員,  
I want to 近所の未解決案件の一覧を見たい,  
So that 散歩や外出に意味と目的が生まれる。

**Acceptance Criteria:**
- [ ] 事件一覧画面に事件 #T-01 が表示される
- [ ] 各事件カードに「事件番号・タイトル・エリアタグ・ステータス」が表示される
- [ ] 一覧は新着順で表示される
- [ ] 3秒以内に表示される
- [ ] スマホ画面で読みやすい文字サイズ（16px 以上）で表示される

*デモ対応: Scene 1（事件配信）*

---

### US-03: 事件の詳細を確認する

As a 匿名捜査員,  
I want to 事件の詳細と他の捜査員の状況を確認したい,  
So that 現場に行く前に違和感の焦点を理解できる。

**Acceptance Criteria:**
- [ ] 事件タイトル・説明・違和感の焦点・捜査の問いが表示される
- [ ] 他の捜査員の選択式回答の集計（例：「B を選んだ捜査員が最多」）が表示される
- [ ] 個人を特定できる情報（ユーザーID・氏名等）は一切表示されない
- [ ] 推理の選択肢（A〜D）が表示される
- [ ] 「捜査報告を投稿する」ボタンが表示される

*デモ対応: Scene 1（事件配信）・Scene 2（現場確認）*

---

### US-04: 捜査報告を投稿する

As a 匿名捜査員,  
I want to 現場で感じた違和感を写真または一文で記録・投稿したい,  
So that その場の感覚を証拠として残し、捜査に貢献できる。

**Acceptance Criteria:**
- [ ] 写真1枚または一文テキストのどちらか一方だけで投稿が完了できる
- [ ] 推理選択肢（A〜D）はオプションで、スキップしても投稿できる
- [ ] 投稿完了まで3タップ以内で完了できる
- [ ] 写真アップロード時、クライアント側 JS で Exif 情報を除去してから Presigned URL で S3 へアップロードする
- [ ] 投稿前に安全ガイドライン（車道に出ない・顔やナンバーが写った写真は投稿しない等）が表示される
- [ ] 投稿完了後、「報告を受け付けました」の確認メッセージが表示される
- [ ] 投稿受付完了は3秒以内

*デモ対応: Scene 3（投稿）*

---

### US-05: 役立っている通知を受け取る

As a 匿名捜査員,  
I want to 自分の報告が役立ったかどうかを知りたい,  
So that 役立っている実感が次の行動につながる。

**Acceptance Criteria:**
- [ ] 次回アプリを開いたとき、未読通知バッジが表示される
- [ ] 通知一覧に「投稿整理完了通知」が表示される（例：「あなたの報告は、有力な現場手がかりとして整理されました」）
- [ ] 有効観察と判定された場合、「有効観察通知」が表示される（例：「あなたの観察は、この種の事件で特に精度が高い傾向があります」）
- [ ] 通知は個人を特定できる情報を含まない
- [ ] 通知をタップすると関連する事件詳細へ遷移する

*デモ対応: Scene 6（役立っている通知）*

---

### US-06: 特命捜査員への予兆を感じる

As a 匿名捜査員,  
I want to 自分がより信頼される捜査員に近づいていることを感じたい,  
So that 匿名で始めても、役に立つ側へ進んでいく実感が得られる。

**Acceptance Criteria:**
- [ ] 有効観察が5件以上かつ類似案件再参照が1件以上の場合、`role_state` が `special_candidate` に遷移する
- [ ] `special_candidate` 遷移時に予兆通知が生成される（例：「街は、あなたのような観察者を必要とし始めています。類似案件の追加検分依頼が届く可能性があります」）
- [ ] UI 上に役職名・バッジ等は表示しない（通知文のニュアンスのみ）
- [ ] `role_state` の値（`anonymous` / `trusted_candidate` / `special_candidate`）は DB に保存される

*デモ対応: Scene 6（役立っている通知）の発展形・MVP では事前データで再現*

---

### US-07: 他の捜査員も同じ違和感を感じているか確認する

As a 匿名捜査員,  
I want to 他の捜査員も同じ違和感を感じているか知りたい,  
So that 自分の感覚が確かめられ、もっと追いたくなる。

**Acceptance Criteria:**
- [ ] 事件詳細画面に選択式回答の集計が表示される（例：「この事件に3人が報告しています」「B を選んだ捜査員が最多」）
- [ ] 集計は個人を特定できない形式で表示される
- [ ] 自分が投稿済みの場合、自分の選択が強調表示される

*US-03 の受容基準と重複する部分があるが、「共感・確認」という動機を明示するために独立したストーリーとして定義*

## 行政担当者（田中さん）のストーリー

### US-08: 確認候補地点の一覧を確認する

As a 行政担当者,  
I want to 住民の観察を地点単位で整理して一覧で見たい,  
So that そのままでは業務に使いにくい生の投稿を、週次レビューで活用できる。

**Acceptance Criteria:**
- [ ] 行政ダッシュボードに確認候補地点の一覧が表示される
- [ ] 一覧は新着順で表示される
- [ ] 各地点カードに「地点ラベル・事件名・危険要因タグ・同種報告件数・優先度分類・最終更新日時」が表示される
- [ ] 投稿者の個人情報（ユーザーID・氏名・行動履歴）は一切表示されない
- [ ] ダッシュボード表示は5秒以内
- [ ] 固定パスワードによる認証を経てアクセスできる

*デモ対応: Scene 5（行政ダッシュボード反映）*

---

### US-09: 確認候補地点の詳細を確認する

As a 行政担当者,  
I want to 事故件数にはまだ表れていない確認候補地点の詳細を把握したい,  
So that 予防的な現地確認の手がかりを得られる。

**Acceptance Criteria:**
- [ ] 地点詳細画面に「地点情報・写真・一文報告の要約・危険要因の詳細・報告の時間帯傾向・行政向け摘要」が表示される
- [ ] 危険要因タグ（視認遅延・死角・歩車動線交差等）が表示される
- [ ] 同種報告件数と時間帯傾向が表示される
- [ ] 即応候補 / 中期確認候補の分類が表示される
- [ ] 写真は Exif 除去済みのものが表示される

*デモ対応: Scene 5（行政ダッシュボード反映）*

---

### US-10: 優先度分類を確認する

As a 行政担当者,  
I want to 即応候補と中期確認候補を分けて見たい,  
So that 一度に全部は対応できないため、優先順位をつけて動ける。

**Acceptance Criteria:**
- [ ] 即応候補の条件: 同種報告3件以上、または危険要因タグに「視認遅延」「死角」が含まれ同種報告2件以上
- [ ] 中期確認候補の条件: 同種報告1件以上で即応候補条件を満たさない
- [ ] 優先度分類はルールベースで自動計算される（AI の自動判定ではない）
- [ ] 一覧画面で優先度分類が視覚的に区別できる

---

### US-11: レビュー状態を記録する

As a 行政担当者,  
I want to 確認済み・現地確認予定・対応不要などのレビュー状態を残したい,  
So that 週次レビューの経緯を追えるようにしたい。

**Acceptance Criteria:**
- [ ] 地点詳細画面でレビュー状態（確認済み・現地確認予定・対応不要）を選択・更新できる
- [ ] レビュー状態の変更履歴が ReviewLog に記録される（変更者タイプ・変更日時・コメント）
- [ ] 更新後、一覧画面のレビュー状態表示に即時反映される

*デモ対応: Scene 5（行政ダッシュボード反映）の操作部分*

## 開発者のストーリー

### US-12: デモ用即時処理を実行する

As a 開発者,  
I want to 投稿から行政ダッシュボード反映までを即時実行したい,  
So that 審査員にループ全体を15分以内で見せられる。

**Acceptance Criteria:**
- [ ] `POST /demo/trigger`（`Authorization: Bearer {DEMO_TOKEN}`）で即時処理が起動できる
- [ ] 処理内容: 投稿整理（AI 要約・分類）→ 地点統合 → ダッシュボード反映 → 通知生成
- [ ] 10秒以内に完了する
- [ ] Step Functions Express Workflow の同期実行で実装される
- [ ] デモトークンは環境変数で管理される

*デモ対応: Scene 4（AI 整理）・Scene 5（ダッシュボード反映）・Scene 6（通知）の連続実行*

---

### US-13: 事前デモデータを投入する

As a 開発者,  
I want to デモ用の事前データをすぐ投入・リセットできたい,  
So that デモ本番前に確実な状態を準備できる。

**Acceptance Criteria:**
- [ ] シードデータスクリプトで Event #T-01・類似報告2件・LocationObservation（observation_count=2）・AISummary を投入できる
- [ ] DynamoDB Local でシードデータの確認ができる
- [ ] デモ後にデータをリセットできる
