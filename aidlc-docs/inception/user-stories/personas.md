# ペルソナ定義 — 町内匿名捜査室

---

## Persona 1: 匿名捜査員（主要ペルソナ）

**代表人物**: 佐藤 正雄（さとう まさお）、68歳

### プロフィール
- 退職後3年、元会社員（製造業の現場管理職）
- 妻と2人暮らし、徒歩圏内に子ども家族が住む
- スマホは普通に使える（LINE・ニュースアプリ・乗換案内）
- 毎朝1時間ほど近所を散歩する習慣がある
- 地域の変化をよく覚えている（「あの角、昔は見通しよかったのに」）

### 動機
- 「ここ、なんか変だ」と感じることはあるが、それを誰かに伝える場がない
- 地域の役に立ちたいが、自治会の会議は敷居が高い
- 自分の記憶や経験が「まだ使える」と感じたい
- 散歩に意味や目的が欲しい

### 行動特性
- 複雑な操作・長い説明は避けたい
- 「失敗したくない」という慎重さがある
- 一度信頼すると継続して使う
- 承認・感謝に敏感（「役に立った」と言われると嬉しい）

### 技術リテラシー
- スマホ操作: 中程度（タップ・スクロール・写真撮影は問題なし）
- アプリ登録: やや苦手（メールアドレス入力・パスワード設定が面倒）
- 通知: 気づく（LINEの通知は毎回確認する）

### サービス導入前後の変化
- **導入前**: ただ散歩し、広報や新聞を読み、町の変化に気づいている。しかしその時間と知識が社会価値に接続されていない。
- **導入後**: 散歩を現場確認として行い、違和感を証拠として提出し、自分の記憶を捜査の補助線として使い、役立っている通知によって次の事件も追いたくなる。

### 「人をダメにする」ポイント
仕事もしていないのに毎日どこかの違和感を追って町を歩いている状態になる。

## Persona 2: 行政担当者

**代表人物**: 田中 恵子（たなか けいこ）、42歳

### プロフィール
- 小規模市の道路管理課・交通安全係
- 担当エリアの生活道路・歩道の維持管理が主業務
- 週次で現場確認リストを作成し、月次で上司に報告
- 住民からの電話・窓口相談を受けることもある

### 動機
- 事故が起きてから対応するのではなく、予防的に確認したい
- 住民の声を拾いたいが、電話・窓口では偏りがある（声の大きい人の意見になりがち）
- 「まだ数字になっていない微差」を把握する手段がない
- 週次レビューの効率を上げたい

### 行動特性
- 業務時間内に素早く確認したい（長時間の操作は避けたい）
- 根拠のある情報を好む（「何件の報告があるか」が重要）
- 自分で判断したい（AI の自動判定には懐疑的）
- 記録を残したい（後から「なぜこの地点を確認したか」を説明できるように）

### 技術リテラシー
- PC・スマホ操作: 高い
- 行政システム: 慣れている（Excel・庁内システム）
- 新しいツール: 導入に慎重だが、使いやすければ受け入れる

### 期待する価値
- 苦情ではなく「確認候補地点」として整理された情報
- 個人情報が含まれない安全なデータ
- 自分でレビュー状態を更新できる（記録が残る）

## Persona 3: 開発者（ハッカソンチーム）

**代表人物**: 開発チームメンバー

### プロフィール
- AWS Summit Japan 2026 ハッカソン参加チーム
- TypeScript・Next.js・AWS に習熟
- 限られた時間（ハッカソン期間）で MVP を構築する

### 動機
- 1地域・1主人公・1事件で成立する MVP を作りたい
- デモで「捜査員が投稿 → AI 整理 → 行政ダッシュボード」のループを15分以内に見せたい
- 匿名捜査員から特命捜査員への進化を段階的に実装したい（MVP では予兆のみ）
- 審査員に技術的な深さと完成度を示したい

### 技術的制約
- Bedrock モデルはローカルモック不可・AWS 実環境を使用
- DynamoDB Local でフロントエンド開発を高速化
- SAM でプロトタイプ後、CDK へ移行予定
