## プロダクト名

町内匿名捜査室

## 一行ピッチ

探偵ナイトスクープ的な「街の小さな依頼」を、ポケモンGO的に歩いて解きにいくサービス。

入口は「匿名捜査員」。続けるうちに、街から名指しで頼まれる「特命捜査員」になってしまう。

扱うのは、町の小さな違和感を捜査するゲームである。

## プロダクトの核

町内匿名捜査室は、街の中にある「まだ事件になっていない違和感」を、AI が未解決案件のような「事件」として再構成し、市民を匿名捜査員として街へ送り出すサービスである。

プレイヤーから見えるのは、街歩き謎解き、現場検分、証拠写真、一文報告、推理、役立っている通知、特命化の予兆である。

裏側では、投稿された観察が AI によって要約・分類・統合され、行政が週次レビューで読める「確認候補地点」として整理される。

このサービスは、表では遊び、裏では観察データへの変換という二重構造を持つ。

## ハッカソンテーマへの応答

AWS Summit Japan 2026 AI-DLC ハッカソンの「人をダメにするサービス」というテーマに対して、次のように応答する。

- 散歩が現場確認になる
- 雑談が聞き込みになる
- 昔の記憶が証拠になる
- 新着事件を見ずにいられなくなる
- 自分の勘が街の役に立ったと知ると、次の事件も追いたくなる

本人にとっては行政協力ではなく、あくまで捜査であり、検分であり、証拠集めである。

その没入が少しずつ日常を侵食し、「仕事もしていないのに毎日どこかの違和感を追って町を歩いている」状態になる。このズレが「人をダメにする」ポイントである。

## 重要な設計思想

### 表は謎解き、裏は観察設計

プレイヤー体験は、行政調査・政策参加・苦情処理ではなく、街歩き謎解きとして成立させる。

UI 文言でも「行政に協力してください」「道路改善に貢献してください」という雰囲気は前面に出さない。

裏側では、AI が投稿を整理し、地点・現象・要因単位に抽象化して行政に返す。

### 個人ではなく場所を見る

事件の対象は、特定の人ではなく、場所・導線・視界・入口・避難路・公共空間などの「町の違和感」である。

扱ってよいもの:

- 見通しの悪い角
- 迷いやすい入口
- 足が止まりやすい避難路
- 入りにくい公共施設
- 人が自然に集まる場所
- 案内表示のわかりにくさ
- 町並みや地形の違和感

扱ってはいけないもの:

- 特定個人の行動
- 特定住宅や世帯
- 犯人探し
- 実名ベースの晒し
- 個人監視
- 特定店舗や施設への一方的な悪評収集
- 犯罪捜査や法執行を連想させる事件

### ポイントより承認

中毒性の核は、ポイント・ランキング・レベルアップではなく、「自分の違和感は間違っていなかった」「自分の観察が有力な手がかりになった」と返ってくる承認である。

### 匿名で始まり、特命へ近づく

ユーザーは実名登録なしで匿名捜査員として始める。

継続するうちに、有効観察が蓄積され、内部状態として信頼候補・特命候補に近づく。ただし MVP では正式な特命捜査員への昇格や辞令発行は実装しない。通知文のニュアンスだけで予兆を出す。

### 1地域・1主人公・1事件に絞る

MVP では世界観を広げすぎない。

1地域、1主人公、1事件、1回の投稿、1回の AI 整理、1回の行政ダッシュボード反映、1回の通知でデモを成立させる。

## 主な利用者像

代表的な利用者は佐藤さん、68歳。

- 退職後3年
- スマホは普通に使える
- 地域を歩く習慣がある
- 昔の道や店、町の変化を覚えている
- 「ここ、なんか変だ」は感じるが、それを誰かに伝える場を持っていない

ターゲット層は主に 60〜75歳。これは遠い他人ではなく、私たち自身の近未来として扱う。

この人は、サービス導入前はただ散歩し、広報や新聞を読み、町の変化に気づいている。しかし、その時間と知識が社会価値に接続されていない。

サービス導入後は、散歩を現場確認として行い、違和感を証拠として提出し、自分の記憶を捜査の補助線として使い、役立っている通知によって次の事件も追いたくなる。

## 行政側の利用者像

MVP で想定する行政側ユーザーは、町村〜小規模市規模の交通安全または生活道路管理を担当する係員である。

課長級、議会説明、複数課横断、実自治体連携は MVP 外とする。

行政に返すのは「苦情」や「通報」ではなく、「確認候補地点」である。

行政に返すもの:

- 地点ラベル
- 危険要因タグ
- 写真
- 一文報告の要約
- 同種報告件数
- 時間帯傾向
- 即応候補 / 中期確認候補の分類
- 行政レビュー状態

行政に返さないもの:

- 投稿者の個人情報
- 投稿者ID
- 投稿者の行動履歴
- 未整理の生投稿テキスト
- 断定的な危険判定

行政判断は人間が行う。AI は整理・要約・分類の補助であり、自動施策決定はしない。

## 最初のMVP事件

### 事件番号

#T-01

### タイトル

誰もぶつかっていないのに、みんな少しだけ身構える角

### 事件の概要

町には、事故が起きていなくても、なぜか身体が先に身構える場所がある。

人は立ち止まらない。けれど、全員がほんの少しだけ歩き方を変えている。

この事件でプレイヤーが追うのは、事故そのものではなく、「見えているつもりで、見えるのが半歩遅れる構造」である。

### プレイヤー向け事件カードの方向性

町の中には、事故が起きていなくても、なぜか身体が先に身構える場所がある。

今回の現場は、そんな「まだ事件になっていない場所」だ。

通り過ぎる人は多い。立ち止まる人はいない。けれど、よく見ると全員がほんの少しだけ歩き方を変えている。

その違和感の正体を検分してほしい。

### 推理選択肢

すべて「ありえる答え」として設計する。B が発見感を持つが、他の選択肢も明確に不正解にはしない。

A. 信号や標識が見えにくい

B. 見えるはずのものが、ひとつだけ遅れて見える

C. 歩行者と自転車の動線が交差している

D. 路面の段差や凹凸がある

### 主人公の投稿例

選択: B

一文報告:

曲がってくる自転車が、見えているつもりで一拍遅れて入ってくる感じがした。

### この事件をMVP本命にする理由

- 一文で違和感と事件性が伝わる
- 表では交通安全調査に見えず、謎解きに見える
- 裏では交通安全・生活道路管理に接続しやすい
- 1地点・1回の検分で成立する
- 身体感覚と記憶が証拠になる
- 高齢層の観察眼や地域記憶と相性がよい

## MVPで示すループ

MVP では次の一連の流れを最小構成で見せる。

1. 違和感の事件化
2. 匿名捜査員への配信
3. 写真・一文報告・選択式推理回答の投稿
4. AI による要約・危険要因分類
5. 地点別の危険要因化・観察統合
6. 行政ダッシュボードへの反映
7. 役立っている通知
8. 特命化の予兆

## 価値仮説

### 仮説1

人は「地域の役に立ってください」と頼まれるより、自分の違和感や記憶が事件解決に効いたと感じる方が、継続的に動く。

検証指標:

- 通知開封率
- 通知後の再投稿率
- 通知後の類似案件参加率

### 仮説2

行政が拾いにくいのは、大きな課題ではなく、まだ数字になっていない微差である。

検証指標:

- 行政担当者が確認候補地点として読めると判断した件数
- `admin_review_status` の更新率

### 仮説3

町の微差を事件として物語化すれば、観察・写真投稿は義務ではなく娯楽になる。

検証指標:

- 投稿完了率
- デモ参加者が「やってみたい」と感じた割合

### 仮説4

遊びとして集められた観察でも、AI で構造化すれば、行政側では確認候補地点や改善候補として読める。

検証指標:

- `AISummary.has_high_signal = true` 率
- `LocationObservation` への統合率
- 行政担当者レビューでの有用判定

## MVPで検証したいこと

1. 町の違和感は、事件として成立するか
2. 利用者は、自分の身体感覚や記憶を証拠として提出したくなるか
3. その投稿は、行政側で確認候補地点として読めるか

## 機能要求の候補

### R-01 事件を生成または提供できること

行政メモ、住民投稿、モック地域情報などをもとに、AI が「街の未解決案件」を事件として生成できること。

MVP では #T-01 の1件をモックとして固定してよい。ただし将来、Bedrock による動的生成へ切り替えられる構造を残す。

最低限の Event フィールド:

- `event_id`
- `case_number`
- `title`
- `description`
- `public_summary`
- `investigation_question`
- `choices`
- `area_tag`
- `status`
- `safety_level`
- `created_at`
- `updated_at`

### R-02 事件一覧を表示できること

ユーザーが新着事件、進行中事件、自分が関与した事件を確認できること。

MVP では事件一覧に #T-01 が表示されること。

### R-03 事件詳細を確認できること

事件タイトル、説明、違和感の焦点、他の捜査員の公開可能な投稿断片、不足している情報、次の一手を表示できること。

他の捜査員の投稿は、一文報告・選択式回答・危険要因タグ程度に限定し、ユーザーIDなど個人特定情報は出さない。

### R-04 捜査報告を投稿できること

ユーザーが事件に対して次を投稿できること。

- テキストメモ
- 写真
- 選択式推理回答

投稿は「写真1枚または一文テキストのどちらか1つ」でも成立する。推理選択肢はスキップ可能にするか、少なくとも負担を軽くする。

### R-05 AI が投稿を整理できること

投稿を AI が要約・分類し、事件単位で整理できること。

最低限必要な整理:

- 要約
- 危険要因または観察要因の分類
- 重複・統合候補
- 次の確認ポイント
- 行政向け摘要
- `has_high_signal`

### R-06 同地点・類似地点の報告を統合できること

同じ地点または似た違和感を持つ投稿を `LocationObservation` としてまとめる。

MVP では高度な地理クラスタリングではなく、選択式回答などから導出した `location_key = {event_id}#{choice_value}` を使う案がある。

### R-07 役立っている通知を返せること

ユーザーの投稿や行動が、事件進展や行政側整理にどうつながったかを通知できること。

通知例:

- あなたの報告が有力な現場手がかりとして整理されました
- あなたの写真が確認候補地点の摘要に使われました
- 類似案件の追加検分依頼が届く可能性があります

### R-08 行政ダッシュボードを提供できること

行政側が確認候補地点を一覧・詳細確認できること。

最低限の表示:

- 地点ラベル
- 事件名
- 写真
- 危険要因タグ
- 要約
- 時間帯傾向
- 同種報告件数
- 即応候補 / 中期確認候補
- レビュー状態

### R-09 匿名捜査員として参加開始できること

実名や過度なプロフィール入力なしで、ユーザーがすぐ事件に参加できること。

ユーザーIDは、メールアドレスまたは localStorage UUID をクライアント側で SHA-256 ハッシュ化して匿名IDとして扱う。ハッシュ化前の値はサーバに保存しない。

### R-10 特命への進化予兆を返せること

MVP では正式昇格までは実装しないが、将来的な特命捜査員化を予感させる通知や文言を返す。

DB 上は `role_state` として `anonymous` / `trusted_candidate` / `special_candidate` を持てるようにする。

### R-11 デモ用即時処理トリガーを持てること

通常はバッチ処理で動く投稿整理・ダッシュボード反映を、デモ時に即時実行できること。

候補:

- `POST /demo/trigger`
- `Authorization: Bearer {DEMO_TOKEN}`
- Step Functions Express Workflow の同期実行
- 10秒以内の完了を目標

### R-12 投稿の公開可否フラグを持てること

Report モデルに `visibility_status` を持たせる。

値:

- `public`
- `hidden`
- `deleted`

MVP では非公開化 UI はなくてもよいが、データモデル上のフラグは最初から持つ。

### R-13 投稿モデレーション状態を持てること

Report モデルに `moderation_status` を持たせる。

値:

- `pending`
- `approved`
- `flagged`
- `rejected`

MVP では最低限のバリデーションでよいが、将来のモデレーションに接続できる構造を残す。

## 非機能要求の候補

### N-01 低コスト・低運用負荷

ハッカソン MVP ではサーバレス中心で、固定費を抑える。

AI 推論は事件生成時と投稿整理時に限定する。画像解析や音声解析は MVP 外。

### N-02 高齢層でも使いやすい UI

- 文字サイズは 16px 以上
- 1画面1目的
- 投稿完了まで3タップ以内
- スマホ中心
- 難しい設定を最初に求めない
- 匿名捜査員としてすぐ事件へ入れる

### N-03 リアルタイム性を過剰に要求しない

AI 整理、地点統合、通知生成、ダッシュボード更新は通常リアルタイムでなくてよい。

通常運用は1日1〜2回のバッチで十分。デモ時だけ即時トリガーで対応する。

### N-04 監査可能性

投稿、AI 要約、通知生成、行政レビュー操作の履歴を追えること。

保存候補:

- ReviewLog
- AISummary
- CloudWatch Logs

AI 入出力、使用モデルID、実行時刻、レビュー状態変更は追跡可能にする。

### N-05 安全配慮 UI

投稿前に安全ガイドラインを表示する。

表示する内容:

- 車道に出ない
- 危険な場所から撮影しない
- 顔や車両ナンバーが写った写真は投稿しない
- 個人を特定しない
- 私有地に入らない
- 危ないと感じたら現地確認を中止する

### N-06 Unit 単位で実装しやすいこと

AI-DLC / Kiro で段階的に実装しやすいように、責務を Unit of Work 単位に分ける。

### N-07 プロンプトをコードと分離する

プロンプトは `prompts/` 配下にファイルとして管理し、コードに埋め込まない。

候補ディレクトリ:

- `prompts/event-generation/`
- `prompts/guardrails/`
- `prompts/summarization/`
- `prompts/tagging/`
- `prompts/notifications/`

### N-08 プライバシー保護

- 実名登録を必須にしない
- ハッシュ化前のメールアドレスやデバイス値はサーバに保存しない
- 行政ダッシュボードには個人単位の行動履歴を表示しない
- 顔や車両ナンバーが明瞭な画像は公開しない
- 写真の Exif 情報を除去する
- 常時位置情報は取得しない

### N-09 レスポンス時間

目標:

- 事件一覧表示: 3秒以内
- 投稿受付完了: 3秒以内
- 行政ダッシュボード表示: 5秒以内
- デモ用即時処理: 10秒以内

## ユーザーストーリー候補

### 匿名捜査員

- 近所の未解決案件を見たい。散歩や外出に意味が欲しいから。
- 事件の現場で感じた違和感をすぐ記録したい。その場の感覚が重要だから。
- 写真と一文だけでも参加したい。複雑だと継続できないから。
- 自分の記憶や昔の風景の違いを証拠として残したい。それが今も価値を持つと感じたいから。
- 違和感の正体に近い選択肢を選んで推理したい。自分の勘を試したいから。
- 自分の報告が役立ったか知りたい。役立っている実感が次の行動につながるから。
- 他の捜査員も同じ違和感を感じているか知りたい。自分の感覚が確かめられるともっと追いたくなるから。
- 自分がより信頼される捜査員に近づいていることを感じたい。匿名で始めても、役に立つ側へ進んでいく実感が欲しいから。
- 実名や個人情報を入力せずにすぐ事件に参加したい。最初のハードルが高いと始める気になれないから。

### 行政担当者

- 住民の観察を地点単位で整理して見たい。そのままでは業務に使いにくいから。
- 事故件数にはまだ表れていない確認候補地点を把握したい。予防的な現地確認の手がかりが欲しいから。
- 危険要因が何に集中しているか知りたい。視認遅延、死角、時間帯差などで対応の種類が変わるから。
- 即応候補と中期確認候補を分けて見たい。一度に全部は対応できないから。
- 確認済み、現地確認予定、対応不要などのレビュー状態を残したい。週次レビューの経緯を追えるようにしたいから。

### 運営者 / モデレーター

- 投稿内容の安全性や公開可否を確認したい。個人情報や不適切画像を防ぐ必要があるから。
- 行政メモや住民の違和感を事件として魅力的に再構成したい。事件として面白くなければプレイヤーが動かないから。
- 個人攻撃や危険行動につながる事件は配信前に除外したい。面白さより安全を優先すべきラインがあるから。
- 投稿を危険要因や観察要因に整理したい。プレイヤー体験と行政活用の両方に必要だから。
- 不適切な投稿を後から非公開にできる手段を持ちたい。配信時には判断できない問題が後で発覚することがあるから。

### 開発者

- 1地域・1主人公・1事件で成立する MVP を作りたい。最小構成で面白さと行政接続の両方を証明したいから。
- 匿名捜査員から特命捜査員への進化を段階的に実装したい。MVP では予兆だけに絞り、後から拡張したいから。
- デモ時に投稿から行政ダッシュボード反映までを即時実行したい。審査員にループ全体を15分以内で見せたいから。

## 匿名から特命への進化仕様候補

MVP では正式な特命捜査員への昇格は実装しない。

実装するのは次の3点のみ。

1. 匿名捜査員として参加できる
2. 有効観察に対して役立っている通知を返す
3. 観察の蓄積に応じて、特命候補の予兆を示す通知を返す

### role_state

- `anonymous`
- `trusted_candidate`
- `special_candidate`

### 状態遷移案

- `anonymous` → `trusted_candidate`: 有効観察3件以上、かつ安全違反なし
- `trusted_candidate` → `special_candidate`: 有効観察5件以上、かつ類似案件再参照1件以上、かつ安全違反なし

MVP では UI 上に役職名を表示しない。通知文の差分だけで進化感を出す。

### 通知文の方向性

anonymous:

- あなたの報告は、有力な現場手がかりとして整理されました

trusted_candidate:

- あなたの観察は、この種の事件で特に精度が高い傾向があります

special_candidate:

- 街は、あなたのような観察者を必要とし始めています。類似案件の追加検分依頼が届く可能性があります

## 安全・倫理方針

### TL;DR

1. 個人ではなく場所を見る
2. 犯罪ではなく違和感を扱う
3. 表は謎解き、裏は観察データ
4. AI 生成事件は承認制にする
5. 投稿は匿名・最小限・非公開化可能にする
6. 行政には地点・要因単位で返す
7. 利用者には無理な現地行動を求めない

### 禁止する行動

- 私有地への無断立入り
- 車道への立入り
- 危険な位置からの撮影
- 無断撮影
- 夜間や悪天候での無理な現地確認
- 不審者的接触

### 禁止する内容

- 顔や車両ナンバーが明瞭に写った写真の公開
- 個人を監視・糾弾する事件
- 実名ベースの晒し
- 個人の行動履歴の公開
- 特定店舗・施設への一方的な評価攻撃
- 行政処分や法執行を匂わせる演出

### 「捜査」という言葉の扱い

使ってよい文脈:

- 街歩き謎解き
- 現場確認
- 違和感の検分
- 証拠写真
- 推理
- 匿名捜査員
- 特命捜査員

避ける文脈:

- 犯人
- 容疑者
- 取り締まり
- 通報
- 逮捕
- 違反摘発
- 監視

「徘徊」など、高齢者や認知症を揶揄しうる語は使わない。

## MVPデモシナリオ候補

15分以内で次を見せる。

### Scene 0 導入

サービスの一行ピッチ、世界観、主人公、本日の事件を説明する。

### Scene 1 事件配信

匿名捜査員向け Web アプリで #T-01 の事件一覧・事件詳細を見せる。

### Scene 2 現場確認

架空の町の交差点写真または動画を使い、視認遅延や身体感覚の違和感を説明する。

### Scene 3 投稿

### Scene 4 AI整理

AI による要約・危険要因分類を見せる。

例:

- 視認遅延
- 歩車動線の交差
- 主体ごとの感じ方の差
- 見通しより認知のズレが主因の可能性

### Scene 5 行政ダッシュボード反映

確認候補地点として表示される。

同種報告が3件以上、または視認遅延タグと2件以上の報告により、即応候補として分類される。

### Scene 6 役立っている通知

匿名捜査員側に通知が返る。

例:

あなたの報告は、有力な現場手がかりとして整理されました。類似案件の追加検分依頼が届く可能性があります。

### Scene 7 締め

町の微差を捜査するゲームであり、その副産物として行政が使える確認候補地点が生まれることを伝える。

## 再Inceptionで必ず検討し直すべき整合注意点

### 1. Exif除去方式の不整合

現在の構想には、写真 Exif 除去について不整合がある。

一部資料では「サーバ側で Exif 除去」としているが、Presigned URL でクライアントから S3 へ直接アップロードする場合、Lambda を経由しないためサーバ側の即時 Exif 除去はできない。

候補は次のどちらか。

案A: クライアント側 JS で Exif を除去してから Presigned URL でアップロードする。

案B: S3 イベントで Lambda を起動し、非同期に Exif 除去済み画像を生成する。

MVP では実装コストが低い案Aが有力。ただし安全性と確実性を重視するなら案Bも検討する。

再Inceptionでは、この方針を明確に一本化すること。

### 2. Phase 1 に Unit 6 / Unit 7 を含めるか

現在の構想では、Phase 1 の最優先 Unit として Unit 1, 3, 4, 5, 9, 11 を挙げている。

しかし、行政ダッシュボードに確認候補地点を出すには LocationObservation の統合処理が必要であり、これは Unit 6 の責務である。

また、MVP デモでは役立っている通知を見せたいが、これは Unit 7 の責務である。

つまり、Phase 1 に Unit 6 / Unit 7 を完全に含めない場合、Unit 5 または Unit 11 がデモ用の最小統合・通知生成を一時的に担当する必要がある。

再Inceptionでは、以下のどちらかを決めること。

- Unit 6 / Unit 7 の最小版を Phase 1 に含める
- Unit 11 のデモ用即時処理に、地点統合と通知生成の最小実装を含める

### 3. Bedrock モデル名とリージョンの再確認

既存構想では Claude 3 Haiku / Sonnet や ap-northeast-1 を候補にしているが、モデル提供状況は変わる可能性がある。

Construction 開始前に、利用可能モデル、リージョン、AWS アカウントのモデルアクセス権限を確認する。

### 4. SAM と CDK の選択

既存構想では AWS SAM または AWS CDK TypeScript のどちらでもよいとしている。

再Inceptionでは、ハッカソン速度、ローカル実行、TypeScript 統一、Step Functions の扱いやすさを踏まえて、どちらを採用するか決める。

### 5. 行政ダッシュボードの認証

MVP は簡易パスワード保護でよいが、どこで実装するかを決める必要がある。

候補:

- Next.js 側で簡易セッション管理
- API 側でヘッダーまたはトークン検証
- デモ時は環境変数ベースの固定パスワード

### 6. 投稿写真の扱い

MVP では実在個人情報や顔・ナンバーが写る画像を扱わない方針。

実装上は、画像解析を入れないなら、UI 注意喚起と運営確認を前提にするしかない。

再Inceptionでは、画像モデレーションを MVP 外にするのか、最低限のチェックを入れるのかを明確にする。

### 7. 「役立っている通知」と「進化予兆」の境界

役立っている通知は MVP 中核だが、特命予兆は中程度の優先度である。

デモ上は両方を1つの通知文に混ぜる案がある。

再Inceptionでは、MVP に必要な通知を以下に分けて定義する。

- 投稿整理完了通知
- 有効観察通知
- 特命予兆通知

### 8. AI-DLC 再実行時の入力制御

新しいディレクトリで AI-DLC をやり直す場合、既存の `docs/` や `aidlc-docs/` を参照させすぎると、再び工程がスキップされる可能性がある。

再実行時は、この `idea.md` のみを初期入力として与える。

必要であれば、既存ドキュメントは「参考資料」ではなく「旧版アーカイブ」として扱い、AI-DLC の各工程で新しく成果物を生成させる。
